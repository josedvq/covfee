---
title: Deploying to a server
---

Deploying covfee for a new study starts from a clone of the external `covfee_project` starter repository. The intended workflow is:

1. clone `covfee_project`
2. modify the project files in that clone
3. test locally against a local checkout of the main `covfee` repository
4. build the Docker deployment from that same project repository

In other words, end-user studies should live in the external `covfee_project` repository, not inside the main `covfee` framework repository.

Deploying that project means building a ready-to-run image and starting a small multi-service stack:

1. the Flask/SocketIO backend
2. the standalone Node Redux store service
3. a static file server for `www/` assets

## 1. Clone the starter project

Start by cloning the external `covfee_project` repository:

```bash
git clone <your-covfee_project-repo-url> myproject
cd myproject
```

Replace `<your-covfee_project-repo-url>` with the actual Git URL of the external `covfee_project` repository. Edit `app.py`, `www/`, and `docker/.env` in that cloned project. Test locally before deploying. The rest of this page assumes you are running commands from that cloned project repository.

## 2. Project configuration

Deployment settings live in `docker/.env`.

Example:

```bash
COVFEE_HOST=my-domain.com
COVFEE_PORT=5000
COVFEE_PROJECT_WWW_URL=https://static.my-domain.com
COVFEE_REDUX_STORE_HOST=store
COVFEE_REDUX_STORE_PORT=5555
COVFEE_DATABASE_PATH=/var/lib/covfee/database.covfee.db

COVFEE_COVFEE_SECRET_KEY=MY_SECRET
COVFEE_ADMIN_USERNAME=admin
COVFEE_ADMIN_PASSWORD=change-me
```

Docker Compose injects the `COVFEE_` variables from `docker/.env` into the build and runtime containers, and Covfee reads them directly from the environment.

## 3. Build-time initialization

Deploy images built from `covfee_project` are expected to:

1. clone the `covfee` repository into the image and install it from that checkout, or copy in a local checkout during development
2. install the Node dependencies for `covfee/shared`, `covfee/client`, and `store`
3. build the production frontend bundles
4. copy the project code, including `app.py`, into the image
5. run `covfee make . --deploy --no-launch`

That means the image already contains an initialized deployment database before the runtime services start.

The `covfee_project` and `openvimo_project` starter repositories each include both a default `docker/Dockerfile` that clones `covfee` from GitHub and a `docker/Dockerfile.local` variant for testing local changes from a sibling `covfee/` checkout. Their `docker/docker-compose.local.yml` overrides switch the `covfee` and `store` services to that local-source build while leaving the other services unchanged.

## 4. Runtime services

At runtime, `covfee start . --deploy` starts only the backend. The Node Redux store now lives in the dedicated `store/` package and should run as a separate service alongside the `www` fileserver. The starter repositories include Docker Compose examples for this split deployment model:

- `covfee_project/docker/docker-compose.yml`
- `openvimo_project/docker/docker-compose.yml`

In those stacks, nginx serves the mounted `www/` directory while the backend serves the application and API routes.

For the default remote-clone workflow, run these commands from the starter-project repository you are using:

```bash
docker compose --env-file docker/.env -f docker/docker-compose.yml build
docker compose --env-file docker/.env -f docker/docker-compose.yml up
docker compose --env-file docker/.env -f docker/docker-compose.yml down
```

For local-source testing with a sibling `covfee/` checkout, use the override file:

```bash
docker compose --env-file docker/.env -f docker/docker-compose.yml -f docker/docker-compose.local.yml build
docker compose --env-file docker/.env -f docker/docker-compose.yml -f docker/docker-compose.local.yml up
docker compose --env-file docker/.env -f docker/docker-compose.yml -f docker/docker-compose.local.yml down
```

In the local-source workflow, the override changes the Docker build context to the monorepo root, so it expects either `covfee_project/` and `covfee/` or `openvimo_project/` and `covfee/` to be sibling directories.

## 5. Mounted data

The deploy model no longer mounts the whole project folder. The only host mount in the starter-repo Compose examples is the `www/` folder attached to the static fileserver service.

The deploy database path is expected to be internal to the image/container, for example `/var/lib/covfee/database.covfee.db`.

## 6. Rebuilding

Whenever you change `app.py`, `docker/.env`, or the bundled project code, rebuild the image so `covfee make . --deploy --no-launch` runs again during the image build. Changes inside `www/` only require restarting the fileserver service if that folder is mounted from the host.
